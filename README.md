# CDEK DevOps Test

## 1. Анализ первопричины

### Исходя из предоставленных логов и метрик, какая проблема является
### корневой, а какие — следствием?

Судя по логам и метрикам, корневых изменений в новой версии бэкенда два, и вместе они создали проблемы:

1. **В `/api/products` переделали с JOIN-запроса на N+1 запросы** Раньше все товары подтягивались одним запросом с JOIN, а теперь сначала идёт запрос за категориями, а потом для каждой категории отдельный `SELECT * FROM products WHERE category_id = $1`. Если в бд, например, 20 категорий, то это 20+ запросов вместо одного, от чего постгресу плохо

2. **В `/api/orders` добавили синхронный вызов Redis** для проверки остатков прямо в основном потоке. Это значит, что каждый запрос на оформление заказа теперь держит соединение с Redis на протяжении всей обработки, а не просто ставит задачу в Celery и возвращает ответ

### Объясните причинно-следственную связь между изменением в коде
### /api/products и ошибками на /api/orders.

1. N+1 запросы в `/api/products` перегружают PostgreSQL - CPU вырастает с 30% до 92%, запросы начинают занимать 3-5 секунд
2. Бэкенд тратит больше времени на каждый ответ, растёт количество параллельных соединений
3. `/api/orders` теперь держит Redis-соединение синхронно на всё время обработки заказа и из-за общей перегрузки соединения висят долго
4. Redis active connections: 5 -> 48 -> 120, лимит 100 превышен
5. Новые запросы к `/api/orders` падают с 500 ошибкой `Redis connection pool exhausted` 
6. Задачи не попадают в Celery, очередь растёт с 12 до 1250 задач

То есть `/api/products` убил PostgreSQL, из-за этого всё замедлилось и накопилось, а синхронный Redis в `/api/orders` стал точкой отказа под нагрузкой и в сумме эти две проблемы положили все

- Ошибки `upstream timed out` в Nginx - это следствие того, что Flask не успевает ответить
- Рост памяти бэкенда (512 MiB до 1.4 GiB) - накапливаются незавершённые запросы
- Очередь Celery на 1250 задач - задачи не создаются, раз `/api/orders` падает

## 2. Сбор дополнительных данных

### Какие ещё данные вы запросите у команды, чтобы подтвердить гипотезу?

- **Сколько категорий** в таблице `products`? Это покажет масштаб N+1 - если 50 категорий, то каждый запрос к `/api/products` делает 50+ запросов к БД
- **Индексы на таблице `products`** - есть ли индекс на `category_id`?
- **Конфигурацию Redis**: текущий `maxmemory-policy`, настройки connection pool на стороне бэкенда
- **Размер таблицы `products`** - `SELECT count(*) FROM products`
- **Логи Celery workers** - что происходит с уже накопившимися 1250 задачами, не зависли ли воркеры
- **Коммит/diff новой версии** - посмотреть конкретные изменения в коде своими глазами

### Какие инструменты мониторинга могли бы помочь быстрее локализовать
### проблему?

- **Prometheus + Grafana** - для метрик CPU/memory/connections в реальном времени с алертами. Если бы был дашборд с Redis active connections, проблему было бы видно ещё в 12:15, когда их стало 48
- **Jaeger** - чтобы увидеть, что конкретно занимает 6-10 секунд в запросе `/api/orders`
- **pgBadger** - анализ медленных запросов PostgreSQL

## 3. Краткосрочные и долгосрочные решения

### Что сделать прямо сейчас, чтобы восстановить работу сервиса? (минимум 2
действия)

**Действие 1 - Откат на предыдущую версию бэкенда**  
Это самый быстрый способ восстановить работоспособность. Так как у нас в GitLab CI есть ручной approval, значит можно откатить деплой

**Действие 2 - Если откат невозможен по какой-то причине: временно увеличить лимит соединений Redis.**  
Это не исправит N+1, но уберёт немедленные 500 на `/api/orders`
```bash
docker exec redis redis-cli CONFIG SET maxclients 500
```

### В долгосрочной перспективе

- **Вернуть JOIN-запрос в `/api/products`**, или использовать `IN` запрос: `SELECT * FROM products WHERE category_id IN (1, 2, 3, ...)` - один запрос вместо N
- **Добавить индекс** на `products(category_id)`, если его нет
- **Вынести проверку стоков в `/api/orders`** из синхронного потока: либо кешировать результат в Redis с TTL, либо делать проверку асинхронно через Celery
- **Добавить кеширование** ответа `/api/products` в Redis с разумным TTL
- **Настроить connection pool** на стороне приложения - сейчас 3 реплики бэкенда, и если у каждой `max_connections=50`, итого 150, что уже превышает лимит Redis 100
- **Добавить нагрузочное тестирование** перед деплоем на прод и этот инцидент выявился бы за пару минут теста

## 4. Метрики и алерты

**Redis active connections > 80**
Лимит 100, порог на 80% даёт время среагировать до отказа. В этом инциденте алерт сработал бы в 12:15, когда было 48 соединений и ещё можно было что-то сделать

**PostgreSQL p95 query duration > 1000 ms**
Нормальные запросы занимают десятки миллисекунд. Если p95 перевалил за секунду - это уже сигнал, что что-то не так с запросами или индексами

**HTTP 5xx error rate > 1% за минуту**
Самый прямой признак того, что с сервисом что-то не так. Лучше смотреть отдельно по эндпоинтам

Дополнительно стоило бы следить за длиной очереди Celery - в этом инциденте она выросла до 1250 задач, что совсем плохо
